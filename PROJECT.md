# 억울함 → 문서 (가칭) — 설계 정리

> **Google AI for Social Good** 해커톤 프로젝트 설계 문서
> 한 줄 정체성: **"억울함을 검색하면, 완성된 공식 문서가 나오는 자판기."**
> 대화 없음. 상황을 고르고 → 한 번 말하고 → 틀린 것만 탭한다.

---

## 1. 무슨 문제를 푸는가

억울한 일(보증금 미반환, 임금체불, 환불 거부, 층간소음, 과태료 등)을 당한 사람이
**법률·행정 문서(내용증명·이의신청서·진정서)를 만들지 못해 권리를 못 찾는** 문제.

- 변호사 비용이 부담되는 사람
- '내용증명' 같은 제도가 **있는지조차 모르는** 사람
- 챗봇 앞에서 **뭘 물어야 할지 몰라 막히는** 사람 (특히 노인·약자)

**Social Good 명분**: 법률·행정 접근권의 격차 해소.

---

## 2. 왜 "그냥 챗봇에 물어보면 되잖아"가 아닌가

이 프로젝트의 존재 이유. AI 모델 자체는 공용품이라 차별점이 아니다.
가치는 **모델 주변**에 있다.

### 2-1. 모트(moat) 4축 — 챗봇이 못 하는 것

| 축                   | 챗봇의 한계                                         |
| -------------------- | --------------------------------------------------- |
| ① 실시간·로컬 데이터 | "일반적으로 이렇습니다"만. 실재하는 법령 검증 못 함 |
| ② 행동(action)       | 알려주기만. 제출 양식·다음 단계까지 안 데려감       |
| ③ 지켜봄(proactive)  | 물어봐야 답함                                       |
| ④ 도달(access)       | 프롬프트 쓸 줄 아는 사람만                          |

### 2-2. AI가 "진짜 필요한" 이유 = 생성(generation)

룰베이스가 절대 못 하는 것은 딱 둘: **(1) 지저분한 자유 입력을 이해**, **(2) 새 글을 생성.**
"약자 대신 격식 있는 문서를 **써준다**"는 생성이 본질이라, "룰베이스로 되잖아" 반박이 원천 불가능하다.

### 2-3. 핵심 방어 논리 (심사용 한 줄)

> "GPT한테 물어봐도 되지 않냐고요? 근데 사람들은 **뭘 물어야 할지를 모르고**,
> 챗봇은 **없는 법 조항을 지어내고**, 텍스트만 주지 **어떻게 제출하는지는 안 알려줘요.**"

## 3. 3대 무기 (전부 UI로 눈에 보이게)

1. **먼저 다 써줌 (elicitation을 객관식으로)** — 백지에서 쌓는 게 아니라 완성 초안에서 깎아냄
2. **법령 그라운딩 (할루시네이션 방지)** — 실재하는 조문만 인용 + 클릭 가능한 출처
3. **제출까지 (action)** — 종이 양식 PDF + "우체국 이렇게" 가이드

## 4. UX 설계

### 4-0. 패러다임: 쌓기(챗봇) → 깎기(우리)

|             | 챗봇                | 우리                  |
| ----------- | ------------------- | --------------------- |
| 시작점      | 빈 입력창           | 이미 완성된 문서 초안 |
| 사용자 역할 | 매 턴 입력(노 젓기) | 확인·수정(방향만)     |
| 인터랙션    | 자유 대화(티키타카) | 상황 고르기 + 탭      |

원칙: **AI가 제안하고, 사용자는 탭으로 채택한다 (propose → tap to accept).**

### 4-1. 레이아웃 3원칙

1. **문서가 항상 주인공** — 분할 화면·채팅 컬럼 없음. 중앙에 A4 문서 한 장.
2. **색 = 상태** — 검정(확정) / 🟡노랑(확인 필요, 탭) / 🔵파랑(법적 근거, 탭하면 원문)
3. **한 화면 = 한 결정**

### 4-2. 사용 흐름 (5단계)

**1단계 · 홈: 상황 찾기 (recognition-first)**

- 태그 필터(#전월세 #돈·환불 #직장 #이웃 #소비자 #계약) + 상황 카드 그리드
- 카드 = 아이콘 + 평서문 상황 + 문서유형 배지 (예: 🏠 "보증금을 안 줘요" → [내용증명])
- 사용자가 **문서를 보낼 수 있다는 사실 자체를 발견** = 정보 격차 해소의 시각화
- 도피구: "제 상황이 없어요 → 🎤 직접 말할게요"

**2단계 · 사연 담기 (자유 입력은 여기 딱 한 번)**

- 큰 마이크 버튼(음성 우선) 또는 텍스트. "무슨 일인지 편하게 말씀하세요."
- 힌트 칩(강요 아님): "말하면 좋아요: 날짜 · 금액 · 몇 번 요청"
- 이후 자유 대화 없음

**3단계 · AI가 완성 초안을 던짐 (핵심 반전)**

- 로딩 스피너 X → 문서가 2~3초에 걸쳐 조립되는 연출
- 검정=확정 / 🟡=AI 추측(확인 필요) / 🔵=법 조항 각주
- 상단: 완성도 미터 + 문서유형 배지 + 강도 슬라이더(정중↔강경)

**4단계 · 확인·수정 (탭 위주, 대화 0)**

- 🟡 칸 탭 → 자유 입력창이 아니라 **객관식 칩** (AI가 맥락 맞춰 생성)
- 고르면 검정 확정 + 완성도 미터 상승 ("노란 거 다 없애기" 게임)
- 🔵 법 조항 탭 → 실제 조문 원문 + 출처 링크
- 강도 슬라이더 → 톤 실시간 재생성

**5단계 · 완성·제출 (끝까지)**

- 실제 우체국 양식으로 렌더링 + [PDF][인쇄][복사]
- "이제 이렇게 하세요": 3부 출력 → 인터넷우체국 접수 링크 → 무응답 시 2차 문서
- "상대는 이렇게 나올 수 있어요" 예상 시나리오

### 4-3. 상황 카드/템플릿/태그 = 진짜 자산

- **상황 팩**: 흔한 억울함마다 `문서 골격` + `필요 정보(객관식 질문셋)` + `관련 법조항 포인터`를 미리 큐레이션
- AI는 이 팩을 사용자 사연에 맞게 **채울 뿐**
- 심사 포인트: "핵심은 AI가 아니라, 억울한 상황을 문서 골격·필요 정보·법 근거로 정리한 '상황 팩'. AI는 채우는 도구, 자산은 큐레이션."

## 5. AI 편집 — 채팅창 없이 AI를 쓰는 법

AI는 "입력창"이 아니라 **"버튼과 재작성 엔진"**으로 들어온다.

1. **객관식 선택지를 AI가 생성** — 칸마다 맥락 맞는 칩
2. **지능형 리플로우 🎯** — 한 칸 바꾸면 단순 치환이 아니라 문서 전체 재작성
   (예: 기간 "1개월"→"6개월" 바꾸면 "상당 기간 경과" 문구 + 지연이자 청구 자동 추가). 룰베이스와 갈리는 지점
3. **문장 골라 → AI 액션 칩** — [더 강하게][더 부드럽게][더 짧게][근거 추가][쉽게]
4. **AI 능동 검토** — "반환 기한을 명시하면 강력해요 [추가]"

### 5-1. 객관식의 함정 → "도피구"를 모든 층에

객관식만 있으면 예외 케이스가 갇힌다. 해법: **객관식은 기본 경로, 자유 입력은 항상 열린 도피구.**
| | 챗봇 | 우리 |
|---|---|---|
| 자유 입력이 | 유일한 시작점(강제) | 항상 열린 도피구(선택) |

- 진입: 상황 없음 → 🎤 직접 말하기
- 칸 편집: 칩에 없음 → ✏️ 직접 입력 (그 슬롯 한정, 무한 백지 아님)
- 문장: 표현 싫음 → 직접 고쳐 쓰기 / 음성 명령
- 자유 입력을 써도 **AI가 파싱해 구조화 JSON 슬롯에 재삽입** → 아키텍처 안 깨짐

원칙: **"쉬운 길은 열어주되, 좁은 길로 몰지 않는다." 90%는 탭, 10%는 언제든 자유.**

## 6. 기술 아키텍처

### 6-1. 척추 4개

1. **문서 = 구조화 JSON** (텍스트 덩어리 X — 이것만은 절대 타협 X)
2. **법령 = law.go.kr API 실시간 fetch + 존재 검증** ("AI 제안 → API 증명")
3. **편집 = 객관식 칩(기본) + 자유 입력 도피구 + 리플로우**
4. **인용은 항상 실재 확인된 출처 링크**

### 6-2. 구조화 JSON 예시

```json
{
  "type": "내용증명",
  "slots": { "deposit": "500만원", "end_date": null, "landlord": null },
  "clauses": [{ "text": "…반환받지 못하였습니다", "cited_law_id": "housing_lease_3_2" }],
  "tone": 0.6
}
```

- `slots`의 `null` = 노란칸(확인 필요) 자동 추적
- 칸 바꾸면 영향받는 `clause`만 재생성(리플로우)
- 각 clause의 `cited_law_id`가 검증된 근거를 붙듦
- **같은 구조 하나가 지능형 편집 + 법령 그라운딩을 동시에 가능케 함**

### 6-3. 할루시네이션 방지 (핵심)

> **모델에게 법을 "기억"시키지 말고 "제공"하라.** (parametric memory ❌ → context injection ⭕)

- **enum 강제**: Gemini structured output에서 `cited_law_id`를 팩 안 ID enum으로 제한
  → 모델이 가짜 법을 **물리적으로 못 뱉음**

```json
{ "cited_law_id": { "enum": ["housing_lease_3_2", "civil_618", null] } }
```

- 공통 철칙: 제공된 것 밖 인용 금지 · 인용은 클릭 가능한 출처 · 불확실하면 비워라(fail-loud) · 면책 문구 기본 표시

## 7. 법령 API 연동 — 실측 결과 (검증 완료 ✅)

**국가법령정보 공동활용 API** (`open.law.go.kr`, 법제처). 2026-07-16 직접 호출로 확인.

### 확인된 사실

| 항목           | 결과                                                                                     |
| -------------- | ---------------------------------------------------------------------------------------- |
| 엔드포인트     | `https://www.law.go.kr/DRF/lawSearch.do` (검색), `lawService.do` (본문) — **live**       |
| 공용 개발키    | **`OC=test` 인증 통과** → 등록 대기 없이 오늘 바로 프로토타입 가능                       |
| 출력 포맷      | ⚠️ **반드시 `type=XML`** — `type=JSON`은 test 키에서 `{}` 깨짐                           |
| 응답 형태      | XML, 한글 태그명 (`법령명한글`, `법령ID`, `공포일자`, `조문` …)                          |
| 검색 품질      | ⚠️ **단순 부분일치** (`민법`→`난민법`이 1등). 원시 검색 못 믿음                          |
| 실서비스 OC    | IP/도메인에 묶임. 우리 공인 IP `222.109.229.137` 등록 필요 → **백엔드 고정 IP에서 호출** |
| 확인된 법령 ID | 주택임대차보호법 `001248`, 민법 `001706`                                                 |

### 검증된 호출 예시

```bash
# 검색 (법령ID 얻기)
curl "https://www.law.go.kr/DRF/lawSearch.do?OC=test&target=law&query=주택임대차보호법&type=XML&display=2"
# 본문 조회 (조문 텍스트)
curl "https://www.law.go.kr/DRF/lawService.do?OC=test&target=law&type=XML&ID=001706"
```

### 그라운딩 전략: "AI가 제안 → API가 검증"

1. AI가 상황 보고 근거 후보를 냄 (법령명 + 조 번호)
2. 그 조문을 law.go.kr API로 실제 조회
3. **존재하면** → 원문 인용 + 출처 링크 (실재 확인)
4. **없으면** → 버림 (지어낸 것 자동 폐기)

→ **API가 신뢰 앵커.** 검색 품질이 약하므로 **조문 포인터(법령명+조번호)는 상황 팩에 큐레이션**,
본문 텍스트·존재 검증만 API 실시간. (조문 텍스트를 코드에 박지 말 것 — 개정되면 틀림)

## 8. 하루 MVP 컷 (ponytail)

- ✅ 상황 카드 **3~4개**만 (보증금·월급·환불) + 태그는 시각용
- ✅ 문서 = **구조화 JSON** (절대 타협 X)
- ✅ 법령 = **OC=test + XML 실시간 fetch + 존재 검증**, 조문 포인터는 큐레이션, `enum` 강제
- ✅ 사연 입력 → Gemini structured output → 초안 + 노란칸
- ✅ 편집 = **리플로우 하나**만 제대로 (나머지 AI 편집은 나중)
- ✅ 객관식 칩 + `✏️직접 입력` 도피구
- ✅ 문서 종이 렌더링 + PDF (브라우저 `print` 공짜)
- ⏸ 나중에: 음성(브라우저 SpeechRecognition) · 강도 슬라이더 · 2차 문서 · 예상 시나리오 · 문장별 액션 칩 · Level 2 시맨틱 검색

### 크리티컬 패스

1. **`OC=test` + XML로 즉시 개발 가능** (등록 불필요 — 리스크 해소됨)
2. 실서비스 전환 시에만 실제 OC + 고정 서버 IP 등록

## 9. 기술 스택

- **프론트**: 웹 (JS/React)
- **AI**: Gemini API (`gemini-2.0-flash`), structured output (JSON schema + enum)
- **법령**: 국가법령정보 공동활용 API (XML 파싱)
- **음성(옵션)**: 브라우저 Web Speech API (네이티브, 무료)
- **PDF**: 브라우저 `print` / `window.print()`
